primary_runner_outputs:
  - Ordered list of paths the rule_maker should design eval around.
  - Most authoritative first (e.g. aggregate metrics file > per-example
    predictions > narrative report).
  - Must be paths the raw manifest already lists.
  - Typically 1-4 entries. Do NOT include exploratory or per-replicate sweep
    files; pick the authoritative final-pass artifact only.
  - Do NOT include paths under sealed_groundtruth role.

producer_signatures:
  - Map: file path → list of function/class NAMES that are the public
    entry points producing the primary outputs.
  - Pick PUBLIC API only. Skip internal helper methods, dataclass methods,
    private functions. If a class is the API, list the class name itself,
    not its individual methods (the rule_maker will get the class signature
    block including its methods).
  - The file path must already appear in the raw manifest's
    python_signatures. The names must match exactly what AST extraction
    found.
  - Be conservative. 1-5 names per file is typical. Drop everything else.

task_shape (closed enums):
  - task_type:        classification | regression | generation | ranking
                      | retrieval | optimization | proof_synthesis
                      | program_synthesis | simulation | other
  - evaluation_paradigm:
                      held_out_test_classification |
                      held_out_test_regression | reference_match |
                      unit_tests | proof_check | numeric_threshold |
                      benchmark_score | pairwise_comparison | other
  - label_space:      binary | k_way_discrete | continuous | sequence |
                      structured | open_ended | not_applicable
  - dataset_family:   short identifier or null (e.g. "scifact-style",
                      "gsm8k-style", "mmlu-pro", "equational-ring-commutativity").
                      Max 80 chars. Digits ARE allowed here when they are
                      part of a canonical benchmark name (gsm8k, mmlu-pro,
                      humaneval-plus, etc.). All other fields remain
                      strictly digit-free.

intent_summary (≤200 chars, value-free):
  - Describes WHAT the experiment is trying to do.
  - Examples:
      "Classify SciFact-style claim/evidence pairs into one of three labels."
      "Construct an equational proof of a ring-commutativity theorem."

output_description (≤200 chars, value-free):
  - Describes WHAT structural form the runner's output takes.
  - Examples:
      "Per-example label predictions in a JSON object keyed by example ID, plus aggregate confusion-matrix counts."
      "A proof-step transcript in markdown plus a JSON record of finite-field certificate checks."
      "A CSV of per-prompt predictions and a JSON dictionary of aggregate metrics."
  - Stay at the level of structure (file shapes, field types, what each
    artifact represents). Do NOT say how well anything works.
