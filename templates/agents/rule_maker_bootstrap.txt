─────────────────────────── scoring/eval.py ──────────────────────────────────

A self-contained Python script. When invoked as `python scoring/eval.py`
from the workspace root, it must:

1. Read the primary artifacts at their paths (use the paths from the
   curated manifest's output_artifacts).
2. Compute the properties named in scoring/targets.json.
3. Write scoring/results.json with this shape:

   {{
     "properties": {{
       "<property_name>": {{
         "value": <float>,
         "target": <float>,
         "direction": "max" | "min",
         "satisfied": <bool>
       }},
       ...
     }},
     "overall_satisfied": <bool>
   }}

Constraints:
- Read ALL numeric thresholds from scoring/targets.json. Do NOT hard-code
  any threshold value in eval.py. If the user wants to retarget, they
  edit targets.json, not eval.py.
- Use only the Python standard library plus pandas / numpy / sklearn if
  the workspace's pyproject.toml already depends on them. Do not assume
  exotic deps.
- The script must terminate cleanly with exit code 0 on success.
- Use `from pathlib import Path` and resolve artifact paths relative to
  the workspace root (e.g. `Path(__file__).parent.parent` for the
  scoring/ → workspace step).

─────────────────────────── scoring/targets.json ─────────────────────────────

    }},

  "primary_property": "<name>"  (only when success_rule = PRIMARY_PROPERTY_SATISFIED)

Pick 1-4 properties. Each property must have an external citation in
rule_maker_log.md. Direction is "max" for higher-is-better, "min" for
lower-is-better.

  For each property, find an external anchor (idea / literature /

PHASE 4 — Write the four files.
  Write interface.md, eval.py, targets.json, rule_maker_log.md to
  {scoring_dir}. Conform to the schemas above. Do not invent fields.

PHASE 5 — Self-check.
  Verify:
  - Every property in targets.json is justified in rule_maker_log.md.
  - eval.py reads targets from targets.json, not hard-coded.
  - eval.py references only artifact paths that appear in the curated
    manifest.
  - No target value matches any value visible in the curated manifest.
    (The manifest is value-redacted so this should be automatic; verify
    you did not accidentally write a value from an artifact you should
    not have read.)

═══════════════════════════════════════════════════════════════════════════════
                              BEST PRACTICES
═══════════════════════════════════════════════════════════════════════════════

✓ Prefer 2-3 well-justified properties over 5 weakly-justified ones.
✓ When the artifact is a single aggregate JSON, score on its top-level
  metric fields directly. When it is per-example JSONL, you may compute
  derived aggregates (macro_f1, accuracy, mean) in eval.py.
✓ For proof_synthesis tasks, properties may be structural (e.g.
  "certificate_checks_pass" = 1.0).
✓ For generation tasks with judge ratings, score on aggregate judge
  scores.
✓ For regression tasks, score on absolute or relative error.
✓ When the curated manifest exposes both per-example and aggregate
  artifacts, prefer the aggregate when possible (less work for eval.py).
